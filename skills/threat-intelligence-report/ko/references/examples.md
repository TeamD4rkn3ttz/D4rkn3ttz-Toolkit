# 대화와 적용 예시

## 예시 1: 유출 데이터 진위 검증

사용자: “포럼에서 한국 기업 고객 DB 3천만 건을 판다는데 샘플이 가짜 같아. 검증하고 X 글 써줘.”

수행:

1. 원 게시물, 샘플, 주장 규모, 최초 확인 시각을 확보한다.
2. 이메일 도메인, 전화번호, 주소, 날짜, 행 분포를 확인한다.
3. 과거 유출·공개 데이터·테스트 데이터와 중복을 찾는다.
4. `가짜`를 먼저 결론내지 않고 반증도 찾는다.
5. `합성/조작 정황` 또는 다른 판정과 신뢰도를 제시한다.
6. 상세 근거를 먼저 보여주고 영문 X 초안을 만든다.
7. 사용자 확인 전 게시하지 않는다.

좋은 결론:

> 샘플에는 example.com 이메일, 규칙적으로 증가하는 전화번호, 확인되지 않는 주소가 반복돼 합성 데이터 정황이 강합니다. 다만 전체 원본을 확인하지 못했으므로 현재 판정은 ‘합성/조작 정황, 신뢰도 중간’입니다.

피할 결론:

> 누가 봐도 100% 가짜다.

## 예시 2: 새 보고서 요약

사용자: “이 랜섬웨어 분석 글 요약해줘. 우리 팀 주간 리포트에 넣을 거야.”

수행:

1. 원문 발행일과 분석 대상 사건일을 확인한다.
2. 침투 벡터, 실행 흐름, 페이로드, 영향, IOC를 분리한다.
3. 공식 또는 다른 1차 자료로 핵심 기술 사실을 교차검증한다.
4. 신규성, 우리 독자에게 중요한 이유, 대응 포인트를 요약한다.
5. 원문에 없는 attribution이나 피해 규모를 만들어내지 않는다.

## 예시 3: 텔레그램 재게시 판단

사용자: “해킹그룹 채널에 성명문이 올라왔어. 새 사건인지 확인하고 게시글로 만들어줘.”

수행:

1. 동일 문구·이미지·링크의 최초 공개를 찾는다.
2. 채널이 공식·사칭·홍보·재게시 계정인지 구분한다.
3. 새 침해 증거가 없으면 `새 공개가 아닌 재게시`라고 전면에 표시한다.
4. 피해를 확대할 원문 데이터는 재배포하지 않는다.

## 예시 4: 정보가 부족한 경우

사용자: “이 캡처만 보고 어느 그룹 공격인지 리포트 써줘.”

응답 방식:

- 캡처에서 직접 관찰 가능한 사실을 먼저 정리한다.
- 그룹명을 확정하지 않는다.
- 추가로 필요한 원 URL, 시각, 파일, 해시, 로그를 한 번에 최소 범위로 요청한다.
- 당장은 `현재 판정`, `가능한 가설`, `정보 공백`, `다음 검증` 형식으로 제공한다.

## 반복 피드백 학습

사용자가 “더 짧게”, “팩트만”, “영문으로”, “첫 줄은 이 형식”, “프로젝트 언급은 마지막”처럼 수정하면 현재 결과에 즉시 반영한다. 동일한 피드백이 여러 사례에서 반복되면 다음 스킬 업데이트 때 문체·템플릿 규칙으로 승격할지 사용자에게 확인한다.
